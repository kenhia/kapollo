# S1 — `vt100`

Nuts-and-bolts writeup for the `vt100` stage (crate `spike-vt100`).

## Approach

- **Render path: `tui-term` (`PseudoTerminal::new(parser.screen())`), not raw `vt100`
  `Screen` cell iteration.** Rationale (R6): `tui-term` 0.3.4 resolves cleanly
  against `ratatui` 0.30 (it depends on the split `ratatui_core`/`ratatui_widgets`
  0.30 crates) and re-exports `vt100` 0.16.2, so there is exactly one emulator in
  the graph. The widget handles grapheme/width-correct cell→`Buffer` blitting for
  free; hand-rolling that from `Screen::cell(row, col)` would have re-implemented
  what `tui-term` already does, with no spike-relevant upside. We keep direct
  access to the `vt100::Screen` for everything the widget does *not* cover
  (scrollback offset, selection text extraction, mode flags).
- Selection highlight is drawn as a post-render overlay: after
  `frame.render_widget(PseudoTerminal, area)`, we walk the normalized content-row
  range, map each back to a visible `y` via `content_to_screen`, and flip
  `Modifier::REVERSED` on the underlying `Buffer` cells with `buf.cell_mut((x, y))`.
- Reused, unit-tested helpers from `spike-support` unchanged: `screen_to_content`,
  `content_to_screen`, `auto_scroll`, `normalize`, `detect_mode`, `osc52_frame`,
  `copy_local`, plus the `PtyShell` reader-thread wrapper.

## Coordinate bridge (the interesting part)

`vt100` models scrollback as an **offset from the bottom** (`set_scrollback(n)`,
`0` = live tail), while `spike-support::coords` models a viewport as a `top_row`
window over an absolute history. We bridge with a fixed `BASE = SCROLLBACK_LEN`:

```text
top_row     = BASE - vt100_scrollback           (top_row = BASE at the live tail)
set_scrollback(BASE - top_row)                  (applied each scroll)
content_row = screen_to_content(top_row, y) = BASE - scrollback + y
```

This let the helper suite drive selection/scroll without modification.

## What worked

- `tui-term` + `ratatui` 0.30 + `vt100` 0.16 compiled and linked with **no version
  pinning gymnastics** — the single hardest pre-spike risk (R1/R6) evaporated.
- The `coords` bridge held: dragging past the top/bottom edge auto-scrolls and the
  selection range tracks correctly because content rows are derived from `top_row`,
  not from screen `y`.
- Alt-screen / child-mouse handover via `detect_mode` **corroborated** by vt100's
  own post-parse flags (`screen.alternate_screen()`,
  `screen.mouse_protocol_mode() != MouseProtocolMode::None`). Using vt100 as the
  authority and `detect_mode` as the streaming hint is robust; either alone is
  weaker. `MouseProtocolMode::None` is the correct "off" variant.
- Explicit-copy model (FR-008/018): left-release only *finalizes* a selection;
  copy happens solely on right-press or Ctrl-C while Active; a second left-press or
  Esc cancels. Maps cleanly onto the `SelectionController` state machine.
- OSC 52 default with `--clipboard=arboard` fallback both wired through one
  `copy_selection` path; `contents_between(sr, sc, er, ec)` yields the selected
  text directly from the screen (scrollback-adjusted).

## What fought back

- **No absolute row index in `vt100`.** Scrollback is bottom-relative, so when new
  output arrives *while the user is scrolled up*, the same physical history line
  shifts position. Our `top_row` is a synthetic absolute coord anchored to `BASE`;
  it stays self-consistent for selection math, but a live selection can visually
  drift if the child emits output mid-drag. A shipping grid model will want a true
  monotonic line id (absolute row counter) that the emulator does not provide here.
  → **Genuine spike finding** feeding the S2/S3 comparison: does the emulator give
  stable absolute coordinates?
- Resize lives on `Screen`, not `Parser`: `parser.screen_mut().set_size(rows, cols)`
  (there is no `Parser::set_size` in 0.16). Minor, but a doc-reading tax.
- Highlight overlay is O(selected cells) touching the `Buffer` after the widget
  renders — fine for a spike, but it bypasses any damage tracking the widget might
  expose. `vt100` has no public dirty/damage API, so partial-redraw optimization is
  not available here (scorecard: damage tracking = weak).

## Scorecard notes (S1 column)

- Render correctness: **Good** — `tui-term` blits the `vt100::Screen` faithfully.
- Grapheme segmentation: **Good** — handled by `vt100` cells + `tui-term` width logic.
- Scrollback API: **OK** — present (`set_scrollback`/`scrollback`) but bottom-relative,
  no absolute row id (see "What fought back").
- Selection primitives: **OK** — `contents_between(row, col, row, col)` gives selected
  text directly; coordinates are visible-relative so we bridge via `coords`.
- Mouse/alt-screen ergonomics: **Good** — `alternate_screen()` +
  `mouse_protocol_mode()` give authoritative routing flags.
- Hyperlinks (OSC 8): **Weak** — no OSC 8 hyperlink accessor found in 0.16.
- Images: **None** — no Sixel/Kitty/iTerm image support.
- Damage/dirty tracking: **Weak** — no public damage API; full redraw each frame.
- API ergonomics: **Good** — small, obvious surface (`Parser::new/process/screen`).
- Build/dep weight: **Good** — light; `tui-term` is thin and shares `ratatui`.
- Maintenance health: **OK** — mature/stable but low recent activity; pinned 0.16.
- `/save` reconstruction: **OK** — `contents_between` over the full grid can
  serialize visible+scrollback text, but lacks absolute line ids for stable anchors.
